# Standard Parts Philosophy

Use this when deciding how much to design from scratch.

## Principle

HTML presentation work becomes more reliable when the agent assembles from standard parts: templates, slide types, chart components, image plans, motion patterns, and verification gates.

The agent should spend its creativity on selection, adaptation, integration, and polishing, not on reinventing every visual system from a blank page.

## Standard Parts

Use these parts deliberately:

- visual system: external template, local template, or project design system
- slide roles: title, section, compare, data, image, timeline, quote, ending
- copy pattern: hook, conflict, evidence, mechanism, implication, action
- image asset: cover hero, section visual, evidence photo, texture, map, generated illustration
- data component: KPI, table, matrix, chart, map, process diagram
- motion pattern: reveal, transition, path draw, counter, tab, toggle
- QA gate: browser, mobile, image crop, console, print/PDF when needed

## Selection Before Construction

Before writing large amounts of HTML, decide:

- which visual system owns the deck
- which slide types are needed
- which pages need images
- which pages need charts or diagrams
- which motion patterns support the talk
- which outputs must be verified

This prevents attractive but incoherent decks.

## Build vs. Reuse

Prefer reuse when:

- the task asks for polish or beauty
- a known template already fits the audience and mood
- the deck needs common layouts
- time is limited
- consistency matters more than novelty

Build custom when:

- the topic needs a unique interactive simulation
- brand rules are strict and no template matches
- data visualization is the main product
- the user asks for a distinctive custom site-like experience

## Human and Agent Roles

The human should choose:

- visual direction
- appetite for boldness
- audience fit
- final taste calls

The agent should handle:

- metadata matching
- preview generation
- content restructuring
- template adaptation
- missing layout extension
- interaction implementation
- browser verification

## Failure Mode

Revise the approach if:

- the deck starts as a blank page despite available strong templates
- the user is forced to describe style only in words without seeing previews
- multiple templates are mixed in one deck
- custom code recreates common components poorly
- verification is skipped because the deck "looks okay" in code
